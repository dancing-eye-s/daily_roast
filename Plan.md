# Plan: Daily Roast 광고 카피 앱 구현

이 계획은 `PRD-fortune-cookie-ad-copy-app.md`를 기준으로 앱을 실제로 만들기 위한 실행 단위다. 각 Phase는 30분~1시간 안에 끝낼 수 있는 과업으로 쪼갠다. 매 Phase는 사용자가 눈으로 확인할 수 있는 산출물 또는 명확한 코드 변경을 남기는 것을 원칙으로 한다. 작업 체크박스는 진행하면서 직접 갱신한다.

## Phase 1. 첫 화면 만들기

**목표:** 어떤 기능이 붙기 전이라도 사용자가 앱의 정체성을 바로 이해할 수 있는 첫 화면을 만든다.

**작업**

- [x] `index.html`에 앱 제목, 짧은 소개 문구, 커피 주전자 영역, 열기 버튼 또는 오브젝트 버튼을 배치한다.
- [x] `styles.css`에 모바일 우선 레이아웃과 기본 비주얼 스타일을 만든다.
- [x] PC/태블릿에서는 중앙 모바일 캔버스만 보이고 좌우 여백이 남도록 화면 폭을 제한한다.
- [x] 레퍼런스를 반영해 강조 메인 컬러와 볼드한 빅 타이포그래피 중심으로 화면 스타일을 변경한다.
- [x] 커피 주전자를 누를 수 있다는 affordance가 보이도록 버튼 상태를 만든다.
- [x] 주전자 클릭 시 컵에 커피가 따라지고 연기 위에 카피가 뜨는 인터랙션으로 변경한다.
- [x] 접근성을 위해 버튼 라벨, `aria-live` 결과 영역을 준비한다.

**완료 기준**

- 브라우저에서 첫 화면이 깨지지 않고 보인다.
- 사용자가 5초 안에 “쿠키를 눌러 오늘의 카피를 보는 앱”임을 이해할 수 있다.
- 기능이 없어도 정적 화면으로 제품의 분위기가 전달된다.

## Phase 2. 샘플 카피 데이터 구조 만들기

**목표:** 실제 대한민국광고대상 수상작 데이터로 확장 가능한 카피 데이터 스키마를 만든다.

**작업**

- [ ] `data/fortuneCopies.js` 또는 기존 데이터 파일에 PRD의 데이터 필드를 반영한 샘플 배열을 만든다.
- [ ] 초기에는 권리 검토가 필요한 실제 카피 원문 대신 테스트용 문구 또는 짧은 공개 확인 문구를 사용한다.
- [ ] `id`, `copy`, `brand`, `campaignTitle`, `awardYear`, `category`, `prize`, `sourceUrl`, `rightsStatus`, `isActive` 필드를 포함한다.
- [ ] 데이터 로딩 실패 시 앱이 멈추지 않도록 기본 빈 배열 처리를 고려한다.

**완료 기준**

- 앱에서 사용할 수 있는 카피 데이터 배열이 존재한다.
- 모든 샘플 데이터가 동일한 스키마를 따른다.
- 권리 상태가 불명확한 데이터는 `pending` 또는 `excerpt_only`로 표시된다.

## Phase 3. 커피 주전자 클릭으로 랜덤 카피 보여주기

**목표:** 사용자가 커피 주전자를 누르면 랜덤 카피 1개가 화면에 표시되게 한다.

**작업**

- [ ] `script.js`에서 활성 상태인 카피만 필터링한다.
- [x] 쿠키 클릭 이벤트를 연결한다.
- [ ] 랜덤으로 1개를 선택해 카피, 브랜드, 캠페인, 수상 정보를 결과 카드에 렌더링한다.
- [x] 결과가 나타날 때 쿠키가 열린 상태로 보이도록 CSS 클래스를 토글한다.

**완료 기준**

- 쿠키를 누르면 문구가 표시된다.
- 새로고침 전까지 클릭 결과가 화면에 유지된다.
- 카피뿐 아니라 최소한 브랜드/캠페인/수상연도 메타데이터가 함께 보인다.

## Phase 4. 하루 1회 제한 구현

**목표:** 같은 날짜에는 같은 카피가 다시 표시되고, 새 카피를 뽑을 수 없도록 한다.

**작업**

- [ ] 로컬 날짜 기준 `YYYY-MM-DD` 키를 생성한다.
- [ ] `localStorage`에 오늘의 카피 ID와 선택 날짜를 저장한다.
- [ ] 재방문 시 저장된 날짜가 오늘이면 같은 카피를 즉시 렌더링한다.
- [ ] 이미 열었을 때 버튼 상태와 안내 문구를 잠금 상태로 바꾼다.

**완료 기준**

- 하루에 한 번만 새 카피가 선택된다.
- 같은 날 새로고침해도 동일한 카피가 표시된다.
- 다음 날짜가 되면 새 카피를 뽑을 수 있는 상태로 돌아간다.

## Phase 5. 다음 쿠키까지 남은 시간 표시

**목표:** 하루 제한이 불편하게 느껴지지 않도록 다음 오픈 가능 시간을 명확히 보여준다.

**작업**

- [ ] 다음 로컬 자정까지 남은 시간을 계산한다.
- [ ] 결과 화면 또는 잠금 상태에 `HH:MM:SS` 카운트다운을 표시한다.
- [ ] 1초 단위로 갱신한다.
- [ ] 자정이 지나면 상태를 자동으로 새 쿠키 열기 가능 상태로 전환한다.

**완료 기준**

- 오늘 이미 열었을 때 다음 오픈까지 남은 시간이 보인다.
- 시간이 0이 되면 새 카피를 뽑을 수 있다.
- 카운트다운이 레이아웃을 밀거나 흔들지 않는다.

## Phase 6. 출처/상세 정보 보기

**목표:** 카피가 어디에서 왔는지 사용자가 확인할 수 있게 한다.

**작업**

- [ ] 결과 카드에 출처 보기 버튼을 추가한다.
- [ ] 클릭 시 상세 모달 또는 확장 패널을 표시한다.
- [ ] 작품명, 광고주/브랜드, 광고회사, 수상연도, 부문, 상격, 출처 링크를 보여준다.
- [ ] 권리 안내 문구와 삭제 요청 가능성을 짧게 표시한다.

**완료 기준**

- 사용자가 카피의 출처와 수상 정보를 확인할 수 있다.
- 출처 링크는 새 탭에서 열린다.
- 상세 영역은 모바일 화면에서 잘 읽힌다.

## Phase 7. 공유 기능 추가

**목표:** 사용자가 오늘의 카피 경험을 외부로 공유할 수 있게 한다.

**작업**

- [ ] Web Share API를 지원하는 환경에서는 네이티브 공유를 실행한다.
- [ ] 지원하지 않는 환경에서는 공유 텍스트를 클립보드에 복사한다.
- [ ] 권리 검토 전 기본 공유 문구는 앱 링크와 작품 메타데이터 중심으로 구성한다.
- [ ] 성공/실패 상태 메시지를 표시한다.

**완료 기준**

- 모바일 브라우저에서 공유 플로우가 작동한다.
- 데스크톱에서는 클립보드 복사 fallback이 작동한다.
- 공유 실패 시 사용자에게 조용한 안내가 표시된다.

## Phase 8. 최근 본 카피 제외 로직

**목표:** 가능한 한 같은 카피가 너무 자주 반복되지 않도록 한다.

**작업**

- [ ] `localStorage`에 최근 본 카피 ID 목록을 저장한다.
- [ ] 랜덤 선택 시 최근 N개 또는 최근 N일 카피를 제외한다.
- [ ] 제외 후 후보가 없으면 전체 활성 목록으로 fallback한다.
- [ ] 저장 목록의 최대 길이를 제한한다.

**완료 기준**

- 데이터가 충분할 때 최근 본 카피가 반복 노출되지 않는다.
- 데이터가 부족해도 앱이 정상 동작한다.
- 로컬 저장 데이터가 과도하게 커지지 않는다.

## Phase 9. 콘텐츠 검수 워크플로 정리

**목표:** 실제 대한민국광고대상 수상작 기반 카피를 안전하게 추가할 수 있는 운영 기준을 만든다.

**작업**

- [ ] 콘텐츠 추가 체크리스트를 문서화한다.
- [ ] 출처 확인일, 출처 URL, 권리 상태, 비활성화 기준을 명시한다.
- [ ] 실제 서비스 노출 가능 상태를 `verified` 또는 별도 플래그로 관리한다.
- [ ] 권리 미확인 카피는 앱 기본 랜덤 후보에서 제외하는 규칙을 구현한다.

**완료 기준**

- 새 카피를 추가할 때 따라야 할 기준이 명확하다.
- 권리 미확인 데이터가 실수로 노출되지 않는다.
- 수동 큐레이션을 계속 확장할 수 있다.

## Phase 10. 반응형/접근성 QA

**목표:** 모바일과 데스크톱에서 화면이 깨지지 않고 기본 접근성을 만족하게 한다.

**작업**

- [ ] 360px, 390px, 768px, 1280px 폭에서 레이아웃을 확인한다.
- [ ] 버튼 포커스, 키보드 조작, 스크린리더 라벨을 점검한다.
- [ ] 색 대비와 텍스트 줄바꿈을 확인한다.
- [ ] 동적 텍스트가 길어져도 카드 밖으로 넘치지 않게 조정한다.

**완료 기준**

- 주요 화면에서 텍스트와 버튼이 겹치지 않는다.
- 키보드만으로 쿠키 열기와 상세/공유 기능을 사용할 수 있다.
- 모바일에서 첫 화면과 결과 화면이 자연스럽게 보인다.

## Phase 11. 기본 분석 이벤트 설계

**목표:** 출시 후 사용자가 실제로 매일 여는지 확인할 수 있는 이벤트 구조를 준비한다.

**작업**

- [ ] 추적할 이벤트 이름을 정의한다.
- [ ] `app_view`, `cookie_open`, `copy_revisit`, `source_open`, `share_click` 이벤트를 기준으로 잡는다.
- [ ] MVP에서는 콘솔 또는 내부 함수 형태로 추상화만 해둔다.
- [ ] 추후 GA4, Plausible, PostHog 등으로 교체 가능하게 만든다.

**완료 기준**

- 이벤트 호출 위치가 코드에 준비되어 있다.
- 분석 도구를 바꿔도 앱 로직을 크게 고치지 않아도 된다.
- PRD의 핵심 지표를 측정할 수 있는 기반이 있다.

## Phase 12. 릴리즈 전 마감 정리

**목표:** MVP를 데모 가능한 상태로 정리한다.

**작업**

- [ ] 불필요한 테스트 문구와 콘솔 로그를 제거한다.
- [ ] README 또는 간단한 실행 안내를 추가한다.
- [ ] 데이터 출처와 권리 주의사항을 문서에 연결한다.
- [ ] 브라우저에서 전체 플로우를 한 번 검증한다.

**완료 기준**

- 앱을 열면 첫 화면부터 오늘의 카피 확인까지 끊기지 않는다.
- 데모용 데이터와 실제 서비스용 데이터의 권리 상태가 구분되어 있다.
- 다음 사람이 이어서 개발할 수 있는 문서가 남아 있다.
